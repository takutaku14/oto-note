# **アマチュア音楽団体支援アプリ 要件定義書**

## **1\. システムの目的**

アマチュア音楽団体（オーケストラ、吹奏楽、合唱、軽音サークルなど）の日常的な活動をデジタル化・一元管理し、運営陣の管理コストを削減するとともに、メンバーへの確実な情報伝達を実現する。

* **初期ターゲットと将来の拡張性**: 初期リリースにおけるメインターゲットは「アマチュアオーケストラ」に特化してUXを最適化するが、システム裏側のデータ構造（パート管理など）は最初から汎用的に設計する。将来的には他の音楽団体へもコードを数行追加するだけで柔軟にターゲットを拡大できるプラットフォーム（SaaS）を目指す。  
* **PWA（Progressive Web App）とプッシュ通知**: スマートフォンのホーム画面に追加可能とし、オフラインキャッシュやWebプッシュ通知機能も活用することで、Webブラウザ特有の安っぽさを排除しネイティブアプリ同等のUXを提供する。

## **2\. 認証・アカウント設計**

システム全体の利便性とセキュリティを両立し、システムエラー時のフェイルセーフを確保するため、認証は目的別に2層に分けて完全に切り離して設計する。

1. **アプリ利用のためのログイン認証（全員必須）**  
   * **目的**: 個人を識別し、パーソナライズされたダッシュボード（自分向けのイベント）を表示するため。  
   * **方法**: Firebase Authenticationによる「Googleログイン（またはAppleログイン等）」をデフォルトとする。※この認証は単なるアプリへの本人確認であり、個人のGoogleドライブ等を操作する権限は一切取得しない。  
2. **外部システム連携のためのAPI認証（幹部のみ・任意）**  
   * **目的**: 団体で使用している外部サービス（Googleカレンダー、スプレッドシート、Google Drive等）への自動同期・書き込みをシステムに代行させるため。  
   * **重要方針**: この連携は\*\*「任意（オプション）」\*\*とする。API連携に興味がない団体や、Google側のエラー発生時でも、アプリのメイン機能（イベント作成・出欠回答・掲示板等）は完全に独立して動作し続けられるフェイルセーフな設計とする。

## **3\. ユーザーロール（権限）定義**

本システムはマルチテナント構造のため、権限は「ユーザーが所属する団体（Membership）単位」で個別に付与される。

* **幹部（管理者）**: 団体設定、API連携設定、メンバー管理（権限・ステータス変更）、シーズン作成・管理、あらゆるイベントの作成、対象者指定。  
* **一般メンバー（平団員）**: 自身が対象のイベント閲覧・回答・タスク消化、掲示板の閲覧・コメント。

## **4\. 機能要件**

### **4.1. マルチテナント管理とシーズン（公演期間）管理機能**

* アカウント1つで複数団体に所属可能。幹部が発行した「招待リンク」から一律「一般メンバー」として参加し、パートは自己申告する。  
* 音楽団体の活動周期に合わせ、「第10回定期演奏会」などの\*\*シーズン（公演期間）\*\*を作成・管理・表示切り替えできる機能。  
* **カウントダウン機能**: シーズンの「本番日」を設定することで、ダッシュボード上に「本番まであと〇日」というカウントダウンを大きく表示し、メンバーのモチベーション向上を図る。

### **4.2. 全団体横断ダッシュボード（マイタイムライン）と統合イベント管理**

* **マイタイムライン（全団体横断表示）**: 複数団体を掛け持ちするメンバーの恩恵を最大化するため、アプリのトップ画面（ホーム）には\*\*「自分が所属するすべての団体の予定とタスク」を時系列で統合した縦スクロール型のタイムライン\*\*を表示する。ダブルブッキングを防止し、個人のスケジュール管理ツールとしての価値を高める。各カードにはどの団体の予定か一目で分かるよう団体カラーのバッジを表示する。  
* **団体別ダッシュボード**: マイタイムラインとは別に、特定の団体（およびシーズン）に絞り込んだ予定や掲示板を確認できる団体別ページも提供する。  
* **イベント責任者の概念**: すべてのイベントにおいて「作成者」とは別に「責任者（Manager）」を設定できる。デフォルトは作成者自身だが、飲み会の宴会幹事やパート内の連絡役など、別のメンバーに責任者権限を委譲できる。  
* **権限制御の明確化**: タスクの完了状況の管理や、支払いイベントにおける「受領確認済」の付与は、**「幹部（システム管理者）」および「そのイベントの責任者」のみが操作可能**とする。  
* **回答アクションと外部同期の仕組み**: メンバーの回答（出欠、支払報告等）は、まずアプリ内データベース（Firestore）を即座に更新する。その後、システムが幹部の顔パス（トークンやAPI経由）を使って裏側で外部システム（スプレッドシート等）へ非同期で連携同期を行う。メンバーにはスプレッドシートへのアクセス権限などを一切求めない。

### **4.3. メンバーグループの一括指定機能**

* イベントの対象者や、掲示板の公開範囲を指定する際、セクション・パート別、システム権限別（幹部/一般）、役職バッジ別などでメンバーを一括選択（ショートカット）できる共通UIを提供する。

### **4.4. スケジュール・タスク機能の統合**

* **スケジュール機能**:  
  * タイムテーブル統合: 1つのイベント内に、時間帯ごとの内容と場所を複数登録できる。  
  * 外部カレンダー連携: 団体カレンダーへの自動同期、個人カレンダーへのワンタップ追加機能。  
* **集金機能（双方向の確認ステップ）**:  
  * 支払いイベントの作成とステータス管理。  
  * メンバーからの「支払報告済」という一方通行の報告に対し、幹部（またはイベント責任者）が銀行口座や手渡しで実際の入金を確認した後、**「受領確認済」フラグを立てる2段階の確認ステップ**を設ける。メンバーの「無事に支払いが完了したか」という不安を払拭し、金銭トラブルを防ぐ。  
* **アンケート機能**: 外部フォーム（Googleフォーム等）のURLを登録したイベントを作成する。メンバーはURLから外部ブラウザを開いて回答し、アプリに戻って「回答済」ボタンを押すことでタスクを完了させる。  
* **タスク管理**: 重要なお知らせ、当番、備品返却などをイベント化し、対象者に「確認済」「完了済」のタスク消化を促す。未完了アラートを視覚的に通知する。  
* **出欠・乗り番統合ビュー（インスペクター向け）**: 外部の乗り番表（編成表）データを読み込み、アプリ内の出欠データと結合表示し、**不足メンバーをハイライトして警告する機能**。

### **4.5. 掲示板・コミュニケーション機能**

* **カテゴリ別スレッド**: 質問、選曲議論、雑談などのカテゴリ分け。シーズンごとにスレッドを管理し、過去シーズンのアーカイブ閲覧も可能とする。  
* **公開範囲（閲覧権限）の指定【NEW】**: スレッド作成時、「幹部のみ」「特定のパートのみ」といったように公開範囲（閲覧可能な対象者）を限定できる機能。これにより、ステータス管理が不要なクローズドな相談や雑談の場を提供する。  
* **純粋なBBSへの特化**: 重要連絡の確実な既読管理は「お知らせイベント（確認タスク）」で行うため、掲示板システム自体に複雑な既読・未読の管理機能は持たせない。  
* **シームレスな資料共有**: スレッド内のコメント等からシステム経由でファイルをアップロードし、自動的に外部クラウドストレージ（Google Drive等）へ保存、リンクを発行・共有する機能。

### **4.6. メンバー・組織管理機能（階層型パート・バッジ・ステータス管理）**

* **汎用的な階層型パート選択**: 表記揺れを防ぐため「セクション名 → 楽器/役割名」の階層構造から選択する。（将来の他ジャンル拡張に対応可能な構造）  
* **役職バッジ機能**: 「インスペクター」「パートリーダー」などの任意のテキストタグ（バッジ）をメンバーに複数付与し、名簿でのハイライトや対象者一括指定に利用する。将来的に「バンマス」等への読み替えも自由入力により可能。  
* **メンバー名簿とステータス管理**:  
  * 団体ごとの氏名、パート、ロール、在籍ステータス（現役・休団・退団）の管理。※名簿はシーズンごとではなく、団体全体で最新状態を一つだけ維持する。  
  * **休団者の扱い**: イベント作成時、対象者のデフォルトチェックから除外する配慮を行う（ただし掲示板の閲覧やコメントは引き続き可能とする）。  
  * **退団者の扱い（論理削除）**: 過去の出欠や会計記録の整合性を保つためデータの物理削除は行わず、ステータスを「退団」に変更する。退団者は当該団体への一切のアクセス権を失い、一般名簿や新規イベント対象者から完全に除外される（幹部のみが閲覧・管理可能）。

### **4.7. プッシュ通知と「アプリ内通知センター」**

* **Webプッシュ通知機能 (FCM)**: Firebase Cloud Messagingを活用し、メンバーのスマートフォン（iOS/Android対応）のロック画面へネイティブアプリ同様のプッシュ通知を送信する。  
  * **即時通知**: 幹部が「重要なお知らせ」等を作成した際、またはイベント責任者に別のユーザーが指名された際に即時通知を送信する（フロントエンド・トリガー）。  
  * **自動リマインド通知（将来拡張）**: 出欠締切前日や本番前日等に自動でリマインド通知を送信する（バックエンド導入後に実装）。  
* **個人宛の通知一覧（インボックス / 通知センター）**:  
  * プッシュ通知をスワイプして見逃した場合に備え、アプリ内に個人宛の通知履歴を一覧表示する画面を設ける。通知には既読/未読ステータスを持たせ、タップすると該当のイベント詳細等へ遷移し既読化する。

## **5\. 非機能要件・運用方針**

* **UI/UXデザイン目標（Apple HIG準拠による質感の追求）**:  
  * Webアプリ特有の安っぽさを完全に排除し、iPhoneのネイティブアプリ（純正アプリ）と見紛うレベルの圧倒的な質感とUXをTailwind CSSとReactで本気で実現する。  
  * スマホ表示でのセーフエリア対応、スクロールバウンス（ゴムバンド効果）の適切な制御、タップハイライトの無効化、テキスト選択・虫眼鏡拡大の無効化などを徹底する。  
* **外部API連携のアーキテクチャ進化戦略（コストと拡張性の両立）**  
  * **初期フェーズ（MVP）**: Firebaseの完全無料枠（Sparkプラン）を維持し開発スピードを最速化するため、外部サービス（スプレッドシートへの出欠書き込み等）との連携は **Google Apps Script (GAS) をWeb API化してフロントエンドから直接通信する方式**を採用する。  
  * **★GAS連携の必須セキュリティ防衛策**: リスクを最小化するため、以下の3点を実装必須とする。  
    1. **POST専用設計**: GAS側は書き込み処理のみとし、名簿等のデータ読み取り（GET）は絶対に実装しない。  
    2. **簡易APIキーの付与**: リクエストヘッダに合言葉を持たせ、無差別なスパムツールを100%弾く。  
    3. **UIDバリデーション**: 送信データにFirebaseの userId を含め、GAS側でシート内の名簿と照合し、存在しない見知らぬIDからの不正書き込みを防止する。  
  * **将来フェーズ（SaaS拡大期）**: ユーザー数が増加した段階で、**Cloud Functions for Firebase** （Blazeプラン）を導入し、バックエンド経由でのセキュアなAPI連携および自動リマインド通知（Cron実行）へとシームレスに移行する。  
* **個人情報管理の最小化**: 住所、電話番号などの機微情報はシステム上で保持しない。

## **6\. スコープ外（実装しない機能）**

システムの複雑化を防ぎ、保守負担を最小限にするため、以下の機能は実装しない。

* **個人間メッセージ機能**: 個別の連絡はLINE等の既存チャットアプリを使用する。  
* **大容量ファイルの直接ホスティング**: 楽譜や音源はシステム自体のサーバーには保存せず外部ストレージへ逃がす。  
* **オンライン決済処理**: クレジットカード決済や銀行API連携などは実装せず、あくまで「払った」「受け取った」のステータス管理のみを行う。  
* **乗り番表（編成表）の直接編集機能**: システム上での表計算UI開発を避ける。  
* **複雑なアンケートフォームの直接作成機能**: Googleフォーム等のURL共有による運用とする。  
* **掲示板における既読・未読管理機能**: お知らせイベントのタスク完了機能で代替する。
